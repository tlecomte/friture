Architecture Review

Friture

Deepening opportunities for a real-time audio analyzer (PyQt5 + QML). Goal: testability and AI-navigability through deeper modules and clearer seams.

Generated 2026-06-12 · Branch context: improve-test-suite + integration test WIP

1. Audio ingest module (capture seam)

Strong

Files

  • friture/audiobackend.py
  • friture/audiobuffer.py
  • friture/analyzer.py (wiring)
  • friture/settings.py (device enumeration)

Problem

AudioBackend is a process-wide singleton that imports sounddevice at module load. Any import of settings or backend requires PortAudio. There is no seam between "capture hardware" and "push float frames into AudioBuffer". Tests must patch globals or install system libs. The interface is nearly as complex as the implementation (device lists, JACK/ALSA quirks, callback threading).

Solution

Introduce a small Audio ingest interface: start/stop, list devices, emit (floatdata, input_time, status). Move PortAudio into one adapter. Analyzer and Settings_Dialog depend on the interface, not the singleton. Tests use a SineWaveIngest or replay adapter — same path production widgets use.

Benefits

  • Leverage: one call site wires ingest → buffer; all dock widgets inherit testability.
  • Locality: device quirks and callback threading live behind one module.
  • Tests: CI runs without libportaudio2 for most suites; integration tests drive real widget pipelines via ingest adapter.

2. Dock analysis widget protocol

Strong

Files

  • friture/dock.py, friture/dockmanager.py
  • friture/spectrum.py, levels.py, pitch_tracker.py
  • octavespectrum.py, spectrogram.py, scope.py, …
  • friture/widgetdict.py

Problem

Nine widgets each implement handle_new_data, set_buffer, canvasUpdate, saveState / restoreState with no shared module. Dock duck-types audiowidget with # type: ignore. Understanding "how a dock consumes audio" requires reading every widget. New integration tests duplicate harness wiring per widget.

Solution

Extract a documented Dock analysis widget protocol (or base mixin) defining the ring-buffer contract: which indices to read, frame size invariants, pause/restart semantics. Optionally split each widget into analysis module (pure/numpy) + dock shell (Qt/QML glue).

Benefits

  • Locality: buffer indexing and "new data since last frame" logic in one place.
  • Leverage: AudioHarness becomes the canonical test adapter for all docks.
  • Tests: contract tests on the protocol; widget tests focus on analysis output, not wiring.

3. Spectrum frame analyzer module

Worth exploring

Files

  • friture/spectrum.py (Spectrum_Widget.handle_new_data)
  • friture/spectrum_plot_widget.py
  • friture/audioproc.py (FFT helpers — already partially extracted)

Problem

handle_new_data orchestrates buffer indices, windowing, FFT, HPS peak pick, exponential smoothing, and QML view-model updates in one method. Unit tests on audioproc pass while bugs hide in index arithmetic and smoothing state — poor locality.

Solution

Deepen a Spectrum frame analyzer module: input = mono frame + settings snapshot; output = frequency bins, peak Hz, smoothed curve. Widget becomes a thin adapter pushing results to QML.

Benefits

  • 440 Hz sine → peak label test runs without Qt (already started in widget tests).
  • Smoothing/time-constant changes touch one module.
  • Pattern repeats for octave spectrum (same FFT + band aggregation).

4. Settings bootstrap vs. device discovery

Worth exploring

Files

  • friture/settings.py (Settings_Dialog)
  • friture/analyzer.py (constructs dialog at startup)
  • friture/test/test_app_bootstrap.py (patches required today)

Problem

Settings dialog constructor probes live input devices and can sys.exit when none exist. App bootstrap tests must mock AudioBackend before import. Settings UI and hardware discovery share one shallow module — no seam for headless or CI environments.

Solution

Split device catalog (list/query, injectable) from settings persistence (QSettings, splash flag, buffer sizes). Dialog receives catalog via constructor; fatal "no devices" becomes a policy decision in Analyzer, not in dialog init.

Benefits

  • Bootstrap tests construct real Settings_Dialog with fake catalog.
  • Generator-only or playback-only modes don't need input hardware at dialog creation.

5. Ballistic / smoothing time-constant module

Worth exploring

Files

  • friture/levels.py, longlevels.py
  • friture/spectrum.py, octavespectrum.py
  • friture/ballistic_peak.py (partial extraction)

Problem

Exponential smoothing kernels and peak ballistics are re-derived in multiple widgets with slightly different formulas and state shapes. ballistic_peak.py helps levels but spectrum/octave still carry parallel math — shallow duplication, no shared seam (one adapter today).

Solution

Deepen a Ballistic smoother module parameterized by attack/release or time constants; widgets hold instances per channel or per bin. Interface: update(sample) → value.

Benefits

  • One test suite for IEC-style ballistics; widgets inherit correctness.
  • "Input feels hotter" tuning becomes a single-knob change with predictable cross-widget behavior.

6. Analyzer application shell

Speculative

Files

  • friture/analyzer.py (~580 lines)
  • QML registration, timers, dock manager, level widget, playback

Problem

Friture class is a god-module: QML type registration, signal wiring, timer policy, and window chrome in one file. Hard to test "app starts headless" without constructing most of Qt. Deletion test: removing it removes the app — complexity is real, but interface is as wide as implementation (shallow at app scale).

Solution

Extract Application composition module: given ingest + settings + dock factory, return wired subsystems. Keeps analyzer.py as entrypoint only. Enables phased bootstrap tests.

Benefits

  • Lower risk if done after ingest + dock protocol seams exist.
  • Speculative until smaller deepenings land — large refactor for moderate test gain alone.

Top recommendation

Start with Candidate 1 — Audio ingest module, then Candidate 2 — Dock analysis widget protocol.

Your integration test work (AudioHarness, widget handle_new_data tests) already assumes a push-based buffer path. The missing seam is at capture: once ingest is injectable, the same harness becomes a second adapter alongside PortAudio, and CI no longer couples every test file to libportaudio2. Formalizing the dock protocol (2) then stops each new widget test from re-discovering buffer indexing rules — that is the fastest route to a "real" suite that proves the app works, not just leaf math.
